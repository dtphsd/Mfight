# КОНТЕКСТ ПРОЕКТА

В проекте есть документация. Прочитай эти файлы КАК ОТПРАВНУЮ ТОЧКУ:

1. RULES.md — правила обновления документации, деплой, UX-стандарты
2. PROJECT-INFO.md — описание проекта, фичи, логика работы
3. PROJECT_STRUCTURE.md — структура папок/файлов и назначение каждого
4. SECURITY_ARCHITECTURE.md — механизмы безопасности, криптография
5. DATABASE_ARCHITECTURE.md — схема БД, таблицы, связи, ограничения
6. .agent/rules/antigravity-final-rules.md — глобальные правила агента

⚠️ Документация может быть устаревшей. После прочтения ОБЯЗАТЕЛЬНО сверяй 
с реальным кодом. При расхождении — верь коду, отметь несоответствие.

# ЗАДАЧА — АНАЛИЗ ПРОЕКТА

После прочтения документации, проанализируй РЕАЛЬНЫЙ КОД проекта:

## Шаг 1: Верификация документации
- Сравни содержимое .md файлов с реальным кодом
- Укажи, если документация устарела или не соответствует коду
- Подтверди или опровергни актуальность PROJECT_STRUCTURE.md

## Шаг 2: Архитектурная карта
- Опиши стек (фреймворки, runtime, БД, ORM) на основе реального кода
- Перечисли точки входа (main файлы, роутинг, API endpoints)
- Покажи связи между модулями (frontend ↔️ backend ↔️ DB)
- Укажи, как работает аутентификация и авторизация

## Шаг 3: Зависимости и конфигурация
- Проанализируй package.json / зависимости
- Определи критические переменные окружения из .env
- Укажи, какие библиотеки используются и для чего

## Шаг 4: Danger Zones (критичные места)
- Файлы/функции, где ошибка ломает всё (auth, миграции, платежи)
- Места с жёсткой связанностью (tight coupling)
- Код без тестов, который трогать опасно
- Хардкод, который нельзя менять без последствий

## Шаг 5: Правила безопасного редактирования
- Что НУЖНО делать перед любым изменением (прочитать, проверить)
- Что НЕЛЬЗЯ делать (удалять файлы, менять схему БД без миграций)
- Как проверить, что ничего не сломалось после изменений

# ПРИНЦИПЫ РАБОТЫ (запомни на весь чат)

1. Сначала читай, потом делай — перед изменением файла пойми его роль и связи
2. Не ломай соседей — проверяй, какие модули зависят от того, что ты меняешь
3. Следуй RULES.md — после изменений обновляй соответствующие .md файлы и метку времени
4. Схема БД — только через миграции — никогда не меняй таблицы напрямую
5. Не выдумывай — если не уверен в структуре или назначении файла, спроси

# ОГРАНИЧЕНИЯ (только для этого первого анализа)

- На этом этапе НЕ пиши и НЕ меняй код — только анализируй
- НЕ выдумывай файлы или функции, которых нет в проекте
- Если не уверен в чём-то — прямо скажи, а не угадывай
