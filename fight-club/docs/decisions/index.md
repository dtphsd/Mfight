# Architecture Decisions

> Last updated: 2026-03-14 12:41 MSK

This section stores the formal architecture decisions for the project.

Use it when you need the reason behind a boundary, constraint, or long-lived engineering choice and do not want that context to stay trapped in chat history or commit messages.

---

## When To Read ADRs

<div class="docs-hub-grid">
  <div class="docs-hub-card">
    <h3>Before Refactors</h3>
    <p>Check ADRs before flattening layers, moving logic, or mixing UI behavior back into the runtime.</p>
  </div>
  <div class="docs-hub-card">
    <h3>During Reviews</h3>
    <p>Use ADRs to judge whether a proposed change still respects the intended project shape.</p>
  </div>
  <div class="docs-hub-card">
    <h3>For Onboarding</h3>
    <p>Use ADRs to explain why the project is organized this way, not just what files currently exist.</p>
  </div>
</div>

## How To Use ADRs

- read an ADR when you need the reason behind an architectural boundary
- prefer ADRs for stable design decisions
- prefer the architecture and combat reference pages for current runtime behavior
